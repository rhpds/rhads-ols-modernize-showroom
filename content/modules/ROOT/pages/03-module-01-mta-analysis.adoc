= Module 1: Analyzing legacy applications with MTA
:source-highlighter: rouge
:toc: macro
:toclevels: 1

Your manager at ACME Corp just assigned you to lead the application modernization initiative. "We need to migrate dozens of legacy Java applications to OpenShift," she says. "Start by analyzing what we're dealing with -- I need to know the risks and effort required."

ACME needs to understand the scope and complexity of their legacy application portfolio before committing to migration timelines. You've heard that Migration Toolkit for Applications can provide AI-powered analysis, but you need to experience it firsthand to understand its capabilities.

In this module, you'll experience how MTA provides comprehensive risk assessment and migration recommendations powered by AI.

* Configure and run MTA analysis on legacy Java applications

* Organize applications into migration waves based on complexity

== Exercise 3: Run MTA risk assessment

With the application registered, you'll now run an AI-powered analysis to understand migration complexity and identify risks.

=== Steps

. From the Applications inventory, select the `petclinic-app` application.

. Click **Analyze** to start the analysis wizard.

. Configure the analysis targets:
+
* **Target platform**: Select `Containerization`
* **Additional targets**: Select `OpenShift` and `Cloud readiness`
+
This tells MTA to analyze the application for containerization and OpenShift deployment readiness.

. Configure advanced options:
+
* **Source mode**: `Source code`
* **Enable automated tagging**: Checked
* **Include known libraries**: Checked

. Review the analysis settings and click **Run analysis**.

. Monitor the analysis progress:
+
The analysis will take a few minutes to complete. You can see real-time progress as MTA examines the source code, dependencies, and configuration files.

. Wait for the analysis to complete -- the status will change to "Completed" when ready.

== Exercise 4: Review analysis and recommendations

Now comes the crucial part -- understanding what MTA discovered about ACME's legacy application and what actions are needed for successful migration.

=== Steps

. Click on the completed analysis to open the **Analysis report**.

. Review the **Risk assessment summary**:
+
MTA categorizes findings by risk level (Low, Medium, High). Each risk level indicates the effort required to resolve issues.
+
* **Low risk**: Simple changes, minimal effort
* **Medium risk**: Moderate refactoring needed
* **High risk**: Significant code changes or architectural decisions required

. Explore the **Issues** section:
+
Click through the different issue categories to see specific findings. MTA provides:
+
* Description of the issue
* Affected files and line numbers
* Recommended remediation actions
* Effort estimation

. Review **Technologies detected**:
+
MTA identifies all frameworks, libraries, and technologies used in the application. This helps understand dependencies and potential compatibility issues.

. Examine the **Dependencies** analysis:
+
Check for outdated libraries, security vulnerabilities, or components that need updating for OpenShift compatibility.

. Note the **Story points** estimate:
+
MTA provides an effort estimation in story points. This helps ACME plan migration timelines and resource allocation.
+
Record the total story points for this application -- you'll use this for migration wave planning.

=== Verify

Confirm you understand the analysis results:

✓ Risk assessment shows categorized findings
✓ Issues are clearly documented with remediation guidance
✓ Story points provide effort estimation
✓ You can navigate to specific code locations with issues

== Exercise 5: Identify migration wave opportunities

Based on the analysis results, you'll now organize applications into migration waves. This strategic planning is crucial for ACME's large-scale modernization effort.

=== Steps

. Return to the **Applications** inventory.

. Review the risk and story points for analyzed applications.

. Create migration waves based on complexity:
+
* **Wave 1 (Quick wins)**: Low-risk applications with minimal story points -- prove the process works
* **Wave 2 (Moderate complexity)**: Medium-risk applications -- build team expertise
* **Wave 3 (Complex migrations)**: High-risk applications -- tackle after gaining experience

. In MTA, you can tag applications to organize them:
+
. Select `petclinic-app` and add tags:
+
* Add tag: `migration-wave-1` if low risk
* Add tag: `migration-wave-2` if medium risk
* Add tag: `migration-wave-3` if high risk

. Document your migration wave strategy:
+
Based on the analysis, note which applications should be migrated first and why. Consider:
+
* Business criticality
* Technical complexity
* Resource dependencies
* Team expertise available

=== Verify

Your migration wave planning is complete when:

✓ Applications are categorized by migration complexity
✓ Tags organize applications into logical waves
✓ Migration sequence is based on data-driven risk assessment
✓ Effort estimates inform project planning
