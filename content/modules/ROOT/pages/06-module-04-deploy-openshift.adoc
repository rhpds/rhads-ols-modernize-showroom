This module takes a very direct "inner loop" developer sandbox approach to building and deploying a fully migrated coolstore application that you helped migrate in the previous modules.  It uses a Quarkus extension to enable a command-line driven maven approach to seamlessly build and deploy the application to an OpenShift cluster.  Configurable environment variables allow for lots of application use case and infrastructure specfic scenarios.

In general, Quarkus is an advanced Java framework and provides many ways to build and deploy applications:

* Source to Image aka **s2i** is a Red Hat OpenShift approach that pulls in a git repository, builds the application package inside a container in OpenShift, creates an image container with the application, and then optionally deploys it to OpenShift as a running service.
* OpenShift Pipelines, often referred to as Tekton, provides a cloud native kubernetes hosted approach to building applications through task steps joined into a pipeline that may also include scanning, signing, and other checks.
* Docker/Podman builds are also supported by Quarkus and different native and jar based application build approaches are included in example docker/containerfile artifacts in a Quarkus project.
* Quarkus has additional extensions that can be added into projects that create generated kubernetes manifests and provide other cloud native capabilities for the Quarkus deployment.
* Beyond what is listed here other systems for building and deploying OpenShift compliant applications can be swapped in such as CNCF buildpacks.

Please talk with your lab instructors about these various approaches and also how OpenShift GitOps, and various packaging approaches such as helm can be added into your CI/CD pipeline implementation for enterprise level support.

=== Verify


Many times users of OpenShift may want to confirm the application is running successfully on OpenShift through mostly cli calls.

Here are some examples.

. Get the application URL:
+
[source,bash,subs="attributes"]
----
# Get the external route URL
oc get route coolstore -o jsonpath='{.spec.host}'

# Store URL in variable
APP_URL=$(oc get route coolstore -o jsonpath='{.spec.host}')
echo "Application URL: https://$APP_URL"
----

. Access the application:
+
Open the URL in your browser or test via curl:
+
[source,bash,subs="attributes"]
----
# Test the application endpoint
curl -k https://$APP_URL/q/health

# Test the main application page
curl -k https://$APP_URL/
----

. View application logs:
+
[source,bash,subs="attributes"]
----
# Get logs from the application pods
oc logs -l app.kubernetes.io/name=coolstore --tail=50
----
+
[source,bash,subs="attributes"]
----
# Check pod health
oc get pods -l app=coolstore
+
# Verify readiness and liveness probes
oc describe pod -l app=coolstore | grep -A 5 "Liveness\|Readiness"
+
# Test health endpoint
curl -k https://$APP_URL/q/health | jq .
----


✓ Deployment shows replicas ready
✓ Pods are in "Running" status
✓ Health checks pass successfully
✓ Application is accessible via HTTPS route
✓ Application functionality works correctly
